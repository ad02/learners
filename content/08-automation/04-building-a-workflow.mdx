# Building Your First Workflow

Time to put everything together! In this lesson, we'll walk through building a real automation workflow from start to finish.

## The Scenario

Imagine you run a small business website. People fill out a contact form, and you want the following to happen **automatically**:

1. The form submission triggers the workflow
2. AI summarizes the message (so you get the key points at a glance)
3. A draft blog post is created in WordPress (if it's a testimonial or FAQ)
4. Your team gets notified on Slack

Right now, you might do all of this **by hand** — reading each message, copying text, creating posts, sending messages. That takes 10-15 minutes per submission. With automation, it takes **zero minutes** because the computer does it for you.

## Step 1: Choose Your Trigger

Every workflow starts with a trigger. In this case, our trigger is:

> **"When a contact form is submitted"**

In most automation tools, this uses a **webhook** — a special URL that receives data when someone submits the form. You learned about webhooks in the API module, so this should feel familiar!

You paste the webhook URL into your contact form settings, and now every submission automatically sends data to your automation tool.

**What you should see after this step:** When you test the webhook, you should see data appear in your automation tool that looks like:

```json
{
  "name": "Jane Doe",
  "email": "jane@example.com",
  "message": "I love your product! It saved me hours...",
  "submitted_at": "2026-04-10T10:30:00Z"
}
```

If you see this data, the trigger is working. If nothing appears, double-check that the webhook URL is correct in your form settings.

## Step 2: Add an AI Step

Next, we want AI to summarize the message. This is where **Claude** comes in.

The automation tool sends the form message to Claude with a prompt like:

> "Summarize this customer message in 2 sentences. Identify if it's a question, testimonial, or complaint."

Claude reads the message and sends back a clean summary. No copy-pasting required.

**What you should see after this step:** Claude returns something like:

```json
{
  "summary": "Customer is expressing satisfaction with the product and its time-saving features.",
  "type": "testimonial"
}
```

## Step 3: Add a WordPress Action

Now we take Claude's summary and use it to **create a draft post in WordPress**.

The automation tool connects to your WordPress site using the REST API (which you also learned about!) and creates a new draft post with:

- The customer's message as the content
- Claude's summary as the excerpt
- A category based on the message type

You can review and publish the draft later — or set it to publish automatically if you're feeling bold!

**What you should see after this step:** WordPress returns a response confirming the post was created:

```json
{
  "id": 456,
  "title": "Customer Testimonial from Jane Doe",
  "status": "draft",
  "link": "https://yoursite.com/?p=456"
}
```

## Step 4: Add a Notification

Finally, we send a **Slack message** to your team:

> "New contact form submission from Jane Doe. Summary: Customer is expressing satisfaction with the product and its time-saving features. Type: Testimonial."

Now your whole team knows about the submission instantly, with a clean summary, without anyone having to do anything.

**What you should see after this step:** A message appears in your Slack channel within seconds of the form being submitted. If no message appears, check that the Slack channel name is correct and that your bot has permission to post there.

## Step 5: Test It!

Before turning on any automation, **always test it**. Most tools have a "Test" or "Run Once" button that simulates the trigger and shows you what each step does.

Testing helps you catch problems like:
- Missing data (the form didn't send the right fields)
- Wrong formatting (the WordPress post looks weird)
- Notification issues (the Slack message goes to the wrong channel)

Fix any issues, test again, and then **turn it on**. Your automation is now live!

## Debugging: "My Workflow Isn't Working!"

Something will go wrong. It always does with the first attempt. Here is how to fix the most common issues:

### Problem: The trigger is not firing
- **Check the webhook URL.** Did you copy the correct one? Is it pasted in the right place?
- **Check the method.** Most webhooks expect POST, not GET. Make sure your form is sending a POST request.
- **Check if the workflow is active.** Some tools require you to turn the workflow "on" before the trigger listens.

### Problem: The AI step returns an error
- **Check your API key.** Is it valid? Has it expired? Did you copy the full key?
- **Check the prompt.** Make sure the data from the previous step is correctly mapped into the prompt (using the right variable names).
- **Check rate limits.** If you are testing rapidly, you might hit the API's rate limit.

### Problem: WordPress post is not created
- **Check authentication.** Make sure your username and application password are correct.
- **Check the site URL.** Is it the right domain? Does it include `https://`?
- **Check the content.** WordPress requires at least a title. Make sure it is not empty.

### Problem: Slack notification is missing
- **Check the channel.** Is the channel name spelled correctly (including the `#`)?
- **Check bot permissions.** Your Slack bot needs permission to post in that channel.
- **Check the message format.** Some special characters can break Slack messages.

> **Pro tip:** Test each step individually before connecting them all together. If step 1 works alone and step 2 works alone, connecting them is usually straightforward.

## Testing Strategies

Here is a proven approach for testing workflows:

1. **Test step by step.** Run only the trigger first. Verify data comes through. Then add the AI step and test again. Then add WordPress. Then add Slack. Build and verify one step at a time.

2. **Use test data.** Don't use real customer data while testing. Create a test form submission with obvious fake data like "Test Name" and "This is a test message." That way you can easily identify and delete test posts.

3. **Check the logs.** Every automation tool has an execution log. After each test, check the log to see exactly what data went into each step and what came out. This is the fastest way to find problems.

4. **Test edge cases.** What happens if someone submits the form with an empty message? What if the message is extremely long? Good automations handle unexpected input gracefully.

## Versioning Your Workflows

As your workflows evolve, you will want to keep backups. Here are some strategies:

- **Export your workflow.** Most tools let you download the workflow as a JSON file. Save it in a folder on your computer or in Git.
- **Name your versions.** Use names like "Blog Publisher v1", "Blog Publisher v2" so you can tell them apart.
- **Document changes.** Keep a simple note about what changed in each version (e.g., "v2: added error handling for empty messages").
- **Don't delete old versions immediately.** If the new version breaks, you want to be able to go back to the one that worked.

## Try It Yourself — Build a Real Make.com Workflow (100% Free)

No simulator. You're about to build and **run** your first real automation. Total time: ~20 minutes.

**What we'll build:** A webhook that receives a message, sends it to **Groq AI** for summarization, then emails the summary to you.

**What you need:**
- A free Make.com account ([make.com/en/register](https://www.make.com/en/register))
- Your Groq API key from Module 7 ([console.groq.com](https://console.groq.com))
- Your email address

### Step 1 — Create the Scenario

1. Log in to [Make.com](https://www.make.com)
2. Click **Scenarios** (left sidebar) → **Create a new scenario**
3. A blank canvas appears with a big `+` bubble

### Step 2 — Add the Webhook Trigger

1. Click the `+` bubble → search for **Webhooks** → pick **Custom webhook**
2. Click **Add** → name it `contact-form-test` → **Save**
3. **Copy the webhook URL** shown (starts with `https://hook.eu2.make.com/...`)
4. Click **OK** — Make is now listening

### Step 3 — Send Test Data to the Webhook

Keep the Make scenario tab open. In a new browser tab, open DevTools Console (F12) and run:

```js
fetch("PASTE_YOUR_WEBHOOK_URL_HERE", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    name: "Test User",
    message: "I love your product! It saved me 3 hours this week."
  })
});
```

Go back to Make — the webhook should flash green and say "Successfully determined." Click **OK**.

### Step 4 — Add the Groq AI Step

1. Click the `+` next to the webhook bubble → search for **HTTP** → pick **Make a request**
2. Configure it exactly:
   - **URL:** `https://api.groq.com/openai/v1/chat/completions`
   - **Method:** `POST`
   - **Headers:** click **Add item** twice
     - `Authorization` = `Bearer YOUR_GROQ_KEY` (paste your actual key)
     - `Content-Type` = `application/json`
   - **Body type:** Raw
   - **Content type:** JSON (application/json)
   - **Request content:** paste this, using the variable picker to insert `{{1.message}}`:

```json
{
  "model": "llama-3.1-8b-instant",
  "messages": [
    {"role": "user", "content": "Summarize in one sentence: {{1.message}}"}
  ]
}
```

3. Click **OK**

### Step 5 — Add the Email Step

1. Click `+` → search for **Email** → pick **Send an Email**
2. Create a connection (Make uses a free built-in email — no SMTP setup needed)
3. Configure:
   - **To:** your own email address
   - **Subject:** `New submission from {{1.name}}`
   - **Content:** Click into the field, use the variable picker to pick the Groq response → `choices[0].message.content`
4. Click **OK**

### Step 6 — Run It

1. Click **Run once** (bottom-left)
2. Re-run the DevTools fetch from Step 3
3. Watch the 3 bubbles light up green in order
4. **Check your email** — you should see a real summary of the test message, generated by Groq, delivered by Make

### Step 7 — Turn It On

Flip the **Scheduling** toggle (bottom-left) from **OFF** to **ON**. Now the scenario runs forever, every time the webhook receives data. Real production automation — yours.

> **Free tier reminder:** Make gives you 1,000 operations/month. Each scenario run uses 3 ops (webhook + Groq + email). You could run this ~330 times/month for free. Plenty for learning and small projects.

Look at what you've done! A single form submission now triggers **four automated steps**:

1. Data received via webhook

This is the power of automation workflows. Each step is simple on its own, but chained together they save you **hours** of manual work every week.

## Tips for Your Own Workflows

- **Start small** — automate one simple task before building complex workflows
- **Always test** — run your workflow manually before turning it on
- **Add error handling** — what happens if one step fails? Most tools let you add a "fallback" action
- **Document your workflows** — write a quick note about what each one does, so future-you remembers
- **Monitor after launch** — check your workflow execution logs for the first week to catch any issues you missed during testing
- **Set up failure alerts** — add a notification that emails you if any step in the workflow fails
