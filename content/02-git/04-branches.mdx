# Branches

A **branch** is a copy of your project where you can make changes without affecting the main version. Think of it like a **parallel universe** for your code — you can experiment, build features, and even make mistakes, all without touching the version that's working perfectly.

## Why Use Branches?

- **Work on a new feature** without breaking the main project
- **Experiment safely** — if it doesn't work out, just delete the branch
- **Fix a bug** while someone else works on a new feature — no conflicts
- **Multiple people** can work on different things at the same time

## A Real-World Scenario

Imagine you have an automation script that sends welcome emails to new customers. It's working perfectly. Now your boss asks you to add a feature: include the customer's name in the email.

**Without branches:** You edit the working script directly. If you make a mistake, the emails break for everyone while you're fixing it.

**With branches:** You create a branch called `add-customer-name`. You make your changes there. The main version keeps working perfectly. When your new feature is tested and ready, you merge it into the main version. Zero downtime, zero risk.

## Main Branch

Every project has a **main branch** (usually called `main`). This is the official version of your project — the one that works, the one that's deployed, the one that matters.

**Rule of thumb:** The main branch should always work. Never make risky changes directly on main. Always use a branch.

## Branch Commands

### See All Branches

```
git branch
```

**Expected output:**
```
* main
```

The `*` shows which branch you're currently on.

### Create a New Branch

```
git branch add-email-feature
```

This creates a new branch but doesn't switch to it yet. Think of it like writing a label for a new folder but not opening it.

### Switch to a Branch

```
git checkout add-email-feature
```

Now you're working in the new branch. Any changes you make here won't affect `main`.

**Expected output:**
```
Switched to branch 'add-email-feature'
```

### Create and Switch in One Step

```
git checkout -b add-email-feature
```

This is the shortcut most people use — it creates the branch and switches to it in one command.

### See Which Branch You're On

```
git branch
```

```
* add-email-feature
  main
```

The `*` moved to `add-email-feature` — that's where you are now.

## Making Changes on a Branch

Once you're on your branch, work normally:

1. Edit files in VS Code
2. Stage changes: `git add .`
3. Commit: `git commit -m "Add customer name to email template"`

These commits only exist on your branch. The `main` branch is completely untouched.

## Switching Back to Main

```
git checkout main
```

When you switch back, you'll notice something cool: **your changes disappear from the files**. Don't panic! They're still saved on your branch. Switch back to `add-email-feature` and they'll reappear. Each branch has its own version of the files.

## Merging: Bringing Changes Together

When your feature is ready and tested, you **merge** it into main:

```
# First, switch to main
git checkout main

# Then merge your branch into main
git merge add-email-feature
```

**Expected output:**
```
Updating abc1234..def5678
Fast-forward
 email-template.html | 2 +-
 1 file changed, 1 insertion(+), 1 deletion(-)
```

Now your changes are part of the main branch. The feature branch still exists but you can delete it since its work is done:

```
git branch -d add-email-feature
```

## When to Use Branches

| Situation | Branch Name Example |
|-----------|-------------------|
| Adding a new feature | `add-email-automation` |
| Fixing a bug | `fix-duplicate-emails` |
| Experimenting with an idea | `experiment-new-layout` |
| Updating content | `update-welcome-message` |

Notice the naming pattern: **describe what the branch does**, using hyphens, in lowercase. Good branch names make it easy for your team (and your future self) to understand what each branch is for.

## Merge Conflicts: When Two People Edit the Same Thing

Sometimes two branches change the **same line** in the **same file**. When you try to merge, Git doesn't know which version to keep. This is called a **merge conflict**.

Think of it like this: two people are editing the same paragraph of a document. Person A changes the first sentence to "Hello customers!" and Person B changes it to "Dear valued clients!" — which one should Git keep?

When a conflict happens, Git marks the file like this:

```
<<<<<<< HEAD
Hello customers!
=======
Dear valued clients!
>>>>>>> add-email-feature
```

You manually choose which version to keep (or combine them), delete the marker lines, and commit. It sounds scary but it's usually straightforward — just pick the right version and clean up the markers.

> **Pro tip:** Merge conflicts are less common than you'd think, especially when working solo. When they do happen, don't panic. Read the conflict markers carefully, choose the right version, and commit.

## Common Mistakes to Avoid

> **Watch out for:** Making changes on `main` when you meant to be on a branch. Always check which branch you're on with `git branch` before you start editing.

> **Watch out for:** Forgetting to commit before switching branches. If you have uncommitted changes and try to switch, Git might give you an error. Commit or stash your changes first.

> **Watch out for:** Creating branches from branches from branches. For now, always create new branches from `main`. It keeps things simple.

## Try It Yourself

Practice the full branching workflow:

1. **Go to your practice project** (or create one):
   ```
   cd Desktop/git-practice
   ```

2. **Make sure you're on main:**
   ```
   git checkout main
   ```

3. **Create a new branch and switch to it:**
   ```
   git checkout -b add-greeting
   ```

4. **Create a new file:**
   ```
   echo "Hello, welcome to my project!" > greeting.txt
   git add .
   git commit -m "Add greeting message"
   ```

5. **Switch back to main and notice the file is gone:**
   ```
   git checkout main
   ls
   ```
   No `greeting.txt`! It's safely on the other branch.

6. **Merge the branch into main:**
   ```
   git merge add-greeting
   ls
   ```
   Now `greeting.txt` is here on main!

7. **Clean up by deleting the branch:**
   ```
   git branch -d add-greeting
   ```

## Practice Scenario

You're working on a project with two tasks:
1. Fix a typo in the README
2. Add a new automation script

How would you handle this with branches? Think about it, then compare:

- Create `fix-readme-typo`, fix the typo, commit, merge into main
- Create `add-automation-script`, build the script, commit, merge into main

By using separate branches, each change is isolated. If the automation script has problems, the typo fix is already safely in main.

## Why This Matters

Branches are how teams build software without stepping on each other's toes. Even when working alone, branches give you the freedom to experiment without fear. Try something wild — if it doesn't work, just delete the branch and main is untouched. This safety net is what makes professional developers confident and fast.
