# What is Version Control?

Have you ever worked on a document and wished you could go back to an earlier version? That's exactly what **version control** does — but for your entire project.

## The Problem

Without version control, things go wrong fast:

- You make a change and it breaks something — but you can't remember what you changed
- You have files named `report-final.docx`, `report-final-v2.docx`, `report-ACTUALLY-final.docx` — sound familiar?
- Working with others means overwriting each other's work
- Your computer crashes and you lose everything you've been working on

Here's a real-world disaster scenario: imagine you're working on an automation script that sends emails to 500 customers. You make a small change, test it, and it seems fine. You deploy it. But the change accidentally sends the wrong email to every customer. Without version control, you have no way to see exactly what changed or quickly undo it.

**With version control**, you could go back to the exact moment before the bad change in seconds.

## The Google Docs Comparison

If you've ever used **Google Docs**, you've already used a simple version of version control. Google Docs has a feature called **"Version History"** (File > Version History) that lets you:

- See every change ever made to the document
- See who made each change
- Go back to any previous version

**Git** does the same thing, but it's way more powerful:

| Feature | Google Docs | Git |
|---------|------------|-----|
| Track changes | Yes | Yes |
| See who changed what | Yes | Yes |
| Go back to old versions | Yes | Yes |
| Work on two versions at once | No | Yes (branches!) |
| Work offline | No | Yes |
| Works with any file type | No (just docs) | Yes (code, configs, data, etc.) |

## The Solution: Git

**Git** is a version control system. Think of it like a **time machine for your project**. Every time you save a checkpoint (called a **commit**), Git remembers exactly what every file looked like at that moment.

You can always go back to any checkpoint.

### How Commits Work

Think of commits like **saving your game** in a video game. At each save point:

- Git takes a **snapshot** of every file in your project
- You write a short **message** describing what changed (like "fixed the email template" or "added new automation script")
- That snapshot is saved **forever** — you can always come back to it

Here's what a series of commits looks like over time:

```
Commit 1: "Created project with basic files"
  └── The starting point

Commit 2: "Added email sending script"
  └── Now the project has an email script

Commit 3: "Fixed bug where emails sent twice"
  └── Oops, there was a bug — now it's fixed

Commit 4: "Added customer name to email template"
  └── A small improvement
```

If Commit 4 broke something, you can instantly go back to Commit 3. No guessing, no panic.

## Why Every Professional Uses Git

This isn't optional knowledge — it's a fundamental skill:

- **Every tech company** uses Git (Google, Microsoft, Amazon, startups, agencies — everyone)
- **Every open-source project** on GitHub uses Git
- **Automation projects** need version control because one bad change can break everything
- **Collaboration** is impossible without it — two people can't edit the same files without some system to manage changes
- **Your future self** will thank you when you need to understand what changed and why, three months from now

Even if you're working alone, Git protects you from yourself. We all make mistakes — Git makes sure those mistakes are never permanent.

## Key Terms to Remember

- **Repository (repo)** — a project folder that Git is tracking
- **Commit** — a saved snapshot of your project at a moment in time
- **History** — the complete list of all commits (your project's timeline)
- **Diff** — the difference between two versions (what changed)

## Why This Matters

Version control is the safety net under every automation project. When you start building scripts that send emails, update databases, or post to social media automatically, you need a way to track changes and roll back mistakes. Git is that safety net. Learning it now means you'll never lose work or deploy a broken automation without a way to fix it.
