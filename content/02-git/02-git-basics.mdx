# Git Basics

Git has three key actions you'll use every day. Think of it like mailing a letter:

1. **Add** (stage) — put your changes in an envelope
2. **Commit** — seal the envelope with a message about what's inside
3. **Push** — mail the envelope to the post office (GitHub)

## The Staging Area (The Envelope)

Before we jump into commands, let's understand a concept that confuses many beginners: the **staging area**.

When you change a file, Git notices. But it doesn't automatically save that change to history. You have to **choose** which changes to include in your next commit. This choosing process is called **staging**.

Why? Because sometimes you change 5 files, but only 3 of those changes are ready. Staging lets you pick exactly which changes go into each commit.

Think of it like packing boxes for a move:
- You have stuff scattered around the room (your changes)
- You pick up specific items and place them in a box (staging)
- You seal the box and label it (committing)
- You load the box onto the truck (pushing)

## The Commands

### `git init` — Start Tracking a Project

```
git init
```

This tells Git: "Start watching this folder." You only do this once per project. After this, Git will notice every change you make.

**Expected output:**
```
Initialized empty Git repository in C:/Users/YourName/Desktop/my-project/.git/
```

### `git add` — Stage Your Changes

```
git add .                ← Add all changed files
git add filename.txt     ← Add just one specific file
```

The `.` means "everything." Most of the time, `git add .` is what you want. But if you only want to commit some changes, you can add files individually.

**Expected output:** No output means it worked! (Git is quiet when things go well.)

### `git commit` — Save a Snapshot

```
git commit -m "your message here"
```

This saves everything you staged as a permanent snapshot. The `-m` flag lets you write a message describing what changed.

**Expected output:**
```
[main abc1234] your message here
 2 files changed, 15 insertions(+), 3 deletions(-)
```

This tells you: which branch you're on, a short ID for the commit, your message, and how many files/lines changed.

### `git push` — Send to GitHub

```
git push
```

This uploads your commits to GitHub so they're backed up online and others can see them.

**Expected output:**
```
Enumerating objects: 5, done.
Counting objects: 100% (5/5), done.
Writing objects: 100% (3/3), 312 bytes | 312.00 KiB/s, done.
```

## Checking What's Going On

Two commands you'll use constantly to understand your project's status:

### `git status` — What's Changed?

```
git status
```

This shows you:
- **Red files** — changed but not yet staged (not in the envelope yet)
- **Green files** — staged and ready to commit (in the envelope, ready to seal)
- **"Nothing to commit"** — everything is saved

**Expected output:**
```
On branch main
Changes not staged for commit:
  modified:   script.js       ← this file changed (red)

Untracked files:
  new-file.txt               ← this is a brand new file (red)
```

### `git log` — View Your History

```
git log --oneline
```

This shows a compact list of all your commits:

```
abc1234 Added email template
def5678 Fixed formatting bug
ghi9012 Initial project setup
```

Each line is one commit. The letters/numbers at the start are the commit's unique ID. The `--oneline` flag keeps it short — without it, you'd see a lot more detail per commit.

> **Pro tip:** Press `q` to exit the log view if it fills more than one screen.

## Writing Good Commit Messages

Your commit messages are like a diary for your project. Write them for your future self — the person who will look at this history in 3 months and need to understand what happened.

**Bad messages:**
- "stuff" — what stuff?
- "update" — what update?
- "fix" — what was broken?
- "asdfg" — this helps nobody

**Good messages:**
- "Add welcome email template"
- "Fix bug where emails sent twice"
- "Update customer list with new contacts"
- "Remove unused automation script"

A good pattern: **start with a verb** (Add, Fix, Update, Remove, Create) and describe **what** changed and **why**.

## The Full Workflow

Here's the complete workflow you'll follow every time you make changes:

```
# 1. Make your changes (edit files in VS Code)

# 2. Check what changed
git status

# 3. Stage your changes
git add .

# 4. Commit with a message
git commit -m "Add email automation script"

# 5. Push to GitHub
git push
```

That's it! Five steps. You'll do this dozens of times per project, and it becomes muscle memory quickly.

## Common Mistakes to Avoid

> **Watch out for:** Committing without checking `git status` first. Always look at what you're about to commit. You might accidentally include a file with passwords or test data.

> **Watch out for:** Writing vague commit messages. "Fixed stuff" won't help you when you need to find a specific change three months later.

> **Watch out for:** Forgetting to `git add` before `git commit`. If you commit without staging, nothing gets saved. Git will tell you "nothing to commit" — that's your clue.

> **Pro tip:** If you just made a commit and realized you made a typo in the message, you can fix it with `git commit --amend -m "corrected message"`. But only do this if you haven't pushed yet!

## Try It Yourself

Let's create a repository from scratch and make three commits:

1. **Create a project folder and initialize Git:**
   ```
   cd Desktop
   mkdir git-practice
   cd git-practice
   git init
   ```

2. **Create a file and make your first commit:**
   ```
   echo "My first project" > README.md
   git add .
   git commit -m "Create project with README"
   ```

3. **Edit the file and commit again:**
   Open `README.md` in VS Code, add a second line: "This project tracks my learning progress." Save the file, then:
   ```
   git add .
   git commit -m "Add description to README"
   ```

4. **Create another file and commit:**
   ```
   echo "console.log('hello')" > script.js
   git add .
   git commit -m "Add hello world script"
   ```

5. **View your history:**
   ```
   git log --oneline
   ```

You should see three commits listed, newest first. Congratulations — you just used Git like a professional developer!

## Why This Matters

These three commands — `add`, `commit`, `push` — are the backbone of every software project and every automation workflow. Once you're comfortable with this cycle, you can confidently make changes knowing that nothing is ever truly lost. You can experiment freely because Git always has your back.
